# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project status

This is a D&D 5e character sheet app, now in **Phase 3: Character creation**. Postgres + Prisma are wired up, and a `Character` resource exists end-to-end: `GET /api/characters`, `GET /api/characters/:id`, `POST /api/characters`, and `PATCH /api/characters/:id`, backed by a real `characters` table. Race/class/background are normalized into a baseline catalog (`Race`, `CharacterClass`, `Background`) plus per-character selection tables (`CharacterRace`, `CharacterBackground`, `CharacterClassEntry`) — each selection carries a nullable FK to the catalog *and* its own `name`, so a character's race/class/background can drift from (or omit) a catalog match without breaking anything; `CharacterClassEntry` is one-to-many to leave room for multiclassing later, though `POST /api/characters` only ever creates one entry today.

Inventory is normalized the same way, plus more: an `Item` catalog (`GET /api/items`) feeds the inventory editor's "pick from the catalog instead of hand-authoring everything" path, and each character's `InventoryItem` row carries a nullable `itemId` *and snapshots* the catalog item's own fields (weight/cost/description) at add-time — unlike `CharacterRace`/etc., which only drift a `name`, an `InventoryItem` is fully self-contained so a player can freely edit it (e.g. "Club" → "Club +1") with no merge-with-catalog logic anywhere. Weapon/armor/consumable mechanics (damage dice, AC, a consumable's effect roll) live on optional 1:1 `ItemWeaponDetail`/`ItemArmorDetail`/`ItemConsumableDetail` tables (mirrored as `InventoryWeaponDetail`/etc. for the snapshot side) rather than as columns on `Item`/`InventoryItem` directly — `gear` gets no detail table. Dice fields are decomposed (`...DiceCount`/`...DiceFaces`/`...Modifier`) to match `frontend/src/lib/dice.ts`'s `RollSpec` shape rather than a `"1d6"` string, so a future damage/healing-roll feature can read them directly. `Spell`/`JournalEntry` are still **not** separate models — spellcasting and journal entries live as JSON columns on the `Character` row and are round-tripped opaquely; normalizing those is left for a later phase. Inventory mutations beyond simple reads (add/sell/buy/consume + a transaction ledger) are also a later phase — today's `inventory` is read-only from the API's perspective; it's deliberately absent from `PATCH /api/characters/:id`'s schema since a blind full-array replace can't express intent the way the catalog/snapshot model needs. `NewCharacterCard` on the list page now links to `/characters/new`, a guided single-page creation form that stages its draft in `localStorage` until the player presses Save.

A character's `level` and `proficiencyBonus` are never stored directly — they're always derived from `experiencePoints` via the 5e XP table in `backend/src/lib/experience.ts`, and computed fresh into every API response. Don't add `level`/`proficiencyBonus` back as persisted/settable fields; change `experiencePoints` instead. Likewise, `race`/`class`/`subclass`/`background` are not plain `Character` columns — they're read through the selection relations and flattened back into those names by `serializeCharacter`/`serializeCharacterSummary` in `characters.ts` (for API-shape backward compatibility); don't add them back as direct columns, and don't patch them via `PATCH /api/characters/:id` — editing an existing character's selections is left for a future selection-aware endpoint.

## Commands

Run everything via Docker Compose from the repo root:

```
docker compose up --build
```

This starts `db` (Postgres 17, port 5432), `pgadmin` (DB GUI, port 5050), `backend` (Express, port 4000), and `frontend` (Vite, port 5173). `backend`'s container `CMD` runs `prisma generate && prisma migrate deploy && prisma db seed` before starting `tsx watch`, so a fresh `docker compose up --build` ends up with the schema applied and three sample characters seeded automatically — both steps are idempotent (`migrate deploy` no-ops if already applied, `db seed` upserts) so they're safe to run on every container start. Source is bind-mounted with hot reload as before.

Root-level scripts fan out to both workspaces (`backend`, `frontend` — this is an npm workspaces monorepo):

```
npm run dev      # runs each workspace's dev script
npm run lint      # ESLint in each workspace
npm run test      # Vitest in each workspace
npm run build     # production build in each workspace
```

To run a single workspace's commands directly (e.g. for a single test file), `cd` into `backend/` or `frontend/` and use that workspace's `package.json` scripts, e.g. `npx vitest run src/routes/__tests__/health.test.ts` in `backend/`, or `npx vitest run src/api/client.test.ts` in `frontend/`. **`backend`'s tests now need a reachable Postgres** — run `docker compose up db -d` first, and export `DATABASE_URL` (see `backend/.env.example`/`.env`) in the shell running `vitest`.

Both services can also run outside Docker for faster iteration: `npm install` then `npm run dev` inside `backend/` or `frontend/` (the frontend expects the backend reachable at `VITE_API_URL`, default `http://localhost:4000/api`; the backend expects Postgres reachable at `DATABASE_URL`, e.g. via `docker compose up db -d` plus a `backend/.env` with `DATABASE_URL=postgresql://character_sheet:character_sheet@localhost:5432/character_sheet`). Prisma CLI commands (`prisma generate`, `prisma migrate dev`, `prisma db seed`) are run from `backend/` and also need `DATABASE_URL` set.

## Architecture

**Backend** (`backend/src`): an Express app. `app.ts` builds the Express instance (CORS + JSON middleware, mounts routers under `/api`) and is exported separately from `index.ts` (which just reads `PORT` and calls `listen`) so the app can be imported directly into tests without binding a port — see `routes/__tests__/health.test.ts`, `routes/__tests__/characters.test.ts`, `routes/__tests__/reference.test.ts`, and `routes/__tests__/items.test.ts`, all driven with `supertest`. Each resource gets its own router file under `src/routes/`, mounted in `app.ts` (`health.ts`, `characters.ts`, `reference.ts`, `items.ts`). `src/lib/prisma.ts` exports a singleton `PrismaClient` (using the `@prisma/adapter-pg` driver adapter, required by Prisma 7); `src/lib/experience.ts` holds the 5e XP-to-level/proficiency-bonus table and pure functions derived from it, `src/lib/srd.ts` holds the rest of the 5e rules data character creation needs — the alignment list, the 18-skill-to-ability mapping, ability-modifier math, and `deriveCreatedCharacter` (which turns a player's choices plus the resolved race/class catalog rows into a level-1 character's mechanical fields) — and `src/lib/itemDetail.ts` holds the `serializeWeaponDetail`/`serializeArmorDetail`/`serializeConsumableDetail` functions shared by `characters.ts` (inventory rows) and `items.ts` (the catalog), since both need the same nested `weapon`/`armor`/`consumable` wire shape. Together the `lib` rules files (`experience.ts`, `srd.ts`) are the only place 5e rules data is allowed to live (not duplicated on the frontend, not derived ad hoc in routes). `GET /api/reference` (in `reference.ts`) serves the race/class/background catalog plus the alignment list to the creation form; `GET /api/items` (in `items.ts`) serves the equipment catalog to the (future) inventory editor. The Prisma client is generated into `src/generated/prisma` (gitignored — run `npx prisma generate` after a fresh clone or schema change) rather than into `node_modules`.

**Frontend** (`frontend/src`): a Vite + React + TypeScript SPA styled with Tailwind v4 (loaded via the `@tailwindcss/vite` plugin in `vite.config.ts` — there's intentionally no `tailwind.config.js`/`postcss.config.js`; the only Tailwind setup is the `@import "tailwindcss";` line in `src/index.css`). Routing is `react-router-dom`, declared in `App.tsx`: `/` renders `CharacterListPage`, `/characters/new` renders `CharacterCreatePage`, `/characters/:id` renders `CharacterSheetPage` (the static `new` route is registered before the `:id` param route so it can't be swallowed by it). All three pages fetch real data (no more mock data — `src/mock/` has been removed) and render loading/error states alongside the existing empty/not-found states. All backend calls go through `src/api/client.ts`, which reads the API base URL from `VITE_API_URL` — add new backend calls there rather than calling `fetch` directly from components. `CharacterCreatePage` is the app's first use of both `useNavigate` (redirects to `/characters/:id` on first save, replacing history so the stale empty form doesn't linger) and `localStorage` (stages the in-progress draft under the `character-draft:new` key until save succeeds). `src/lib/abilityGen.ts` holds the four ability-score generation methods (4d6-drop-lowest, standard array, point buy, manual) as pure functions, used by the `AbilityScoreEditor` component.

**Docker Compose**: `db` (Postgres) and `pgadmin` (DB GUI), plus `backend` and `frontend`, each of the latter two built from its own `Dockerfile` with the corresponding directory bind-mounted into the container (so edits on the host trigger hot reload) and an anonymous/named volume shadowing `node_modules` (so the container's installed dependencies aren't clobbered by the bind mount). There is no shared root `Dockerfile` — backend and frontend are independent npm packages, each installing its own dependencies; the root `package.json`'s npm workspaces setup is only used for fanning out `dev`/`lint`/`test`/`build` scripts when working outside Docker, not for the container builds.

Race/class/background were already normalized this way (see `prisma/schema.prisma`'s `Race`/`CharacterClass`/`Background` catalog models and `CharacterRace`/`CharacterBackground`/`CharacterClassEntry` selection models); inventory now is too (`Item` catalog, `InventoryItem` snapshot rows, plus `ItemWeaponDetail`/`ItemArmorDetail`/`ItemConsumableDetail` — see schema.prisma's comments on those models for the snapshot-vs-overlay reasoning). That update corrected an earlier guess in this file that inventory wouldn't need a catalog table the way races/classes do — it did, since the goal is letting players pick from a baseline list instead of hand-authoring every item. When a later phase normalizes spells/journal into their own models, re-check whether they need the same catalog treatment rather than assuming `Spell`/`JournalEntry` as bare relations off `Character` — spells arguably do (a spell catalog), journal entries don't (there's no baseline list to pick from). Either way: generate a migration, and add new router files under `backend/src/routes/` following the `characters.ts`/`items.ts` pattern, each paired with a new function in `frontend/src/api/client.ts` and a component under `frontend/src/components/`.
