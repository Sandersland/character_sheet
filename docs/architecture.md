# Architecture

## Backend (`backend/src`)

### Router map — all mounted under `/api` in `app.ts`

*Source of truth: `backend/src/app.ts` mounts — regenerate this table from there if it looks stale.*

**Request pipeline & auth gate (#101).** `app.ts` mounts, in order: security headers → CORS → JSON body → request logger → rate limiters → `healthRouter` → `authRouter` → **`requireAuth`** → every other router → SPA static (optional) → JSON 404 → terminal error handler. The **public allowlist** is exactly `health` + `/api/auth/*` (mounted before `requireAuth`); every router below the gate requires a valid session (401 otherwise — including unknown `/api/*` paths, so existence isn't leaked). Character-scoped routes additionally call `assertCharacterAccess` (see Identity & ownership) for per-owner 403.

| File | Endpoints |
|---|---|
| `routes/health.ts` | `GET /health` — **public** |
| `routes/auth.ts` | `GET /auth/providers`, `GET /auth/:provider/start`, `GET /auth/:provider/callback`, `POST /auth/logout`, `GET /auth/me`, `POST /auth/dev-login` — hand-rolled Google OAuth + PKCE sign-in and opaque session. **Public** (mounted before `requireAuth`). A provider is listed/usable only when its id+secret env pair is set. The provider `startUrl` + post-callback redirect are **origin-relative** (derived from the request host, falling back to `APP_BASE_URL`) so sign-in returns to the right port; the OAuth `redirect_uri` stays `APP_BASE_URL`-based (it must match the registered URI). `POST /auth/dev-login` mints a session for a fixed dev user **without** OAuth — guarded by `ALLOW_DEV_LOGIN`, hard-disabled in production; the headless/worktree path for UI verification (see `seed:verify` in development.md). |
| `routes/characters.ts` | `GET /characters`, `GET /characters/:id`, `POST /characters`, `PATCH /characters/:id`, `DELETE /characters/:id` |
| `routes/reference.ts` | `GET /reference` — race/class/background catalog + alignments + per-class starting equipment options |
| `routes/items.ts` | `GET /items` — item catalog with weapon/armor/consumable detail |
| `routes/hitpoints.ts` | `POST /characters/:id/hp` — batch HP ops |
| `routes/inventory.ts` | `POST /characters/:id/inventory/transactions` (inventory history reads via the unified activity log, `?category=inventory`) |
| `routes/experience.ts` | `POST /characters/:id/experience` — batch XP ops |
| `routes/activity.ts` | `GET /characters/:id/activity`, `POST /characters/:id/events/:batchId/revert` |
| `routes/spells.ts` | `GET /spells` — spell catalog |
| `routes/spellcasting.ts` | `POST /characters/:id/spellcasting/transactions` — batch spell ops |
| `routes/resources.ts` | `POST /characters/:id/resources/transactions` — batch resource/maneuver ops (spend/restore, learn/forget) |
| `routes/conditions.ts` | `POST /characters/:id/conditions/transactions` — apply/remove conditions, set exhaustion |
| `routes/class.ts` | `POST /characters/:id/class/transactions` — post-creation subclass + fighting-style selection + multiclass add-class (`addClass`) |
| `routes/maneuvers.ts` | `GET /maneuvers` — GrantedAbility catalog filtered to `source: "maneuver"` (the 16 SRD Battle Master maneuvers: superiority-die cost + placement/action metadata + save ability); `POST /characters/:id/maneuvers/transactions` — `castManeuver` (spend one superiority die via the shared pool payer, roll it server-side, announce the derived save DC, and self-apply Rally's temp HP) |
| `routes/disciplines.ts` | `GET /disciplines` — GrantedAbility catalog (`source` discriminator) filtered to `source: "discipline"` (Way of the Four Elements: min level + embedded `AbilityCost` + ki-scaled `EffectSpec`); `POST /characters/:id/disciplines/transactions` — `castDiscipline` (spend ki via the shared pool payer, roll the EffectSpec, ki DC save) |
| `routes/shadow-arts.ts` | `GET /shadow-arts` — GrantedAbility catalog filtered to `source: "shadowArts"` (Way of Shadow: the 4 L3 Shadow Arts spells, embedded flat 2-ki `AbilityCost` + non-scaling `EffectSpec`); `POST /characters/:id/shadow-arts/transactions` — `castShadowArt` (spend a flat 2 ki via the shared pool payer, apply concentration + Pass without Trace's +10 Stealth buff) |
| `routes/channel-divinity.ts` | `GET /characters/:id/channel-divinity` — the Channel Divinity options this Cleric/Paladin is entitled to (gated per class/subclass/level server-side; subclass-specific so it's character-scoped, unlike `GET /maneuvers`), each with cost, derived save DC (announce options), and reminder text; `POST /characters/:id/channel-divinity/transactions` — `castChannelDivinity` (spend 1 CD charge via the shared pool payer, apply the option's real side effect: Sacred Weapon's durable `attackRoll` buff, Cloak of Shadows' invisible condition, or announce/advantage/reminder text) |
| `routes/feats.ts` | `GET /feats` — feat catalog |
| `routes/advancement.ts` | `POST /characters/:id/advancement/transactions` — take/remove ASIs and feats |
| `routes/actions.ts` | `POST /characters/:id/actions/transactions` — apply an action's effects in one atomic batch: cast-core actions (`ACTION_CAST_FN`, e.g. Second Wind #420) route through `castAbilityInTx` (pool spend + self-heal), the rest dispatch `ACTION_EFFECT_FN` resource/quantity/heal/buff ops (the action catalog is consumed client-side via `features/session/actionResolvers`). Buff ops (`applyBuff`/`clearBuff`) power Rage: `rage` applies a `while-active` `meleeDamage` buff (scaled by barbarian level), `endRage` clears it. The frontend session turn-hook (`lib/turnHooks.ts`, evaluated in `useTurnActions.handleEndTurn` against the `useTurnState` per-turn attack/damage window) auto-fires `endRage` when a raging turn passes with no attack and no damage — no enemy modeling (any attack counts). |
| `routes/journal.ts` | `POST /characters/:id/journal`, `PATCH /characters/:id/journal/:entryId`, `DELETE /characters/:id/journal/:entryId` — plain-REST; `POST` accepts a full `ENTRY` (date+body) or a fast `NOTE` (body only; date defaults to today, auto-attaches the active session). Reads route through the exported `visibleEntries(db, userId, character)` helper (private-by-default == author's own entries; the `CAMPAIGN`-visible branch is inert until sharing ships). The exported `syncEntryRefs(...)` reconciles `@`-mention backlinks on write, taking a mention-less fast path that skips the character/membership/entity lookups entirely (#489). |
| `routes/sessions.ts` | Campaign-level sessions (#245). Campaign-scoped, `assertCampaignMembership`-gated: `POST /campaigns/:campaignId/sessions` (start; body `{characterId, title?}` — **409** if one is already active), `POST …/sessions/:sessionId/join` / `…/leave`, `POST …/sessions/:sessionId/end` (any member; OWNER may force-end), `GET /campaigns/:campaignId/sessions` (history), `GET …/sessions/:sessionId` (detail w/ participants+events+journals; runs auto-close). Character-scoped reads/writes the SessionPage uses: `GET /characters/:id/sessions/active` (**200 null** when no campaign / no active session — never 404), `GET /characters/:id/sessions/:sessionId`, and `POST …/sessions/:sessionId/combat/{start,end,round}` + `…/roll` (gated on being an active participant). |
| `routes/campaigns.ts` | `GET /campaigns` (the caller's campaigns, each with their `role`), `POST /campaigns` (create + OWNER membership), `GET /campaigns/:id` (members + their characters + caller's `role`), `POST /campaigns/join` (join by invite code as PLAYER, idempotent — an existing OWNER stays OWNER), `POST /campaigns/:id/characters` (attach a character — returns the full serialized character; **409** if the character is already in a different campaign, same-campaign re-attach is a no-op success). Plain-REST shared-campaign backbone (#246); access gated via `assertCampaignMembership` (`lib/auth/access.ts`). |
| `routes/entities.ts` | Campaign entity registry & backlinks (#248), mounted after `campaigns`. `GET /campaigns/:id/entities?q=&type=` (list/search — normalized name+alias match in memory, invalid `type` ignored), `POST /campaigns/:id/entities` (create — any member), `PATCH /campaigns/:id/entities/:entityId` (edit — any member; 404 if not in campaign), `DELETE /campaigns/:id/entities/:entityId` (**OWNER only** → 403 for a PLAYER; cascades refs; 204), `GET /campaigns/:id/entities/:entityId/backlinks` (notes tagging the entity, newest-first `{ entry, characterName }`, filtered through the SAME private-by-default rule as `visibleEntries` so other members' PRIVATE notes never leak). **Visibility (#379):** each entity has a `visibility` (`HIDDEN`\|`REVEALED`, default `REVEALED`). Non-owner reads filter to `REVEALED` on the list, and 404 a `HIDDEN` entity on backlinks + basic-field PATCH (invisible, not just forbidden); the OWNER sees all. Setting/changing `visibility` (on create or PATCH) is **OWNER only** → 403 for a PLAYER; create/edit of basic fields stays member-level. Journal `@`-mention reconciliation (`syncEntryRefs` in `routes/journal.ts`) drops a non-owner's tag of a `HIDDEN` entity so a UUID guess can't materialize a revealing backlink. **Identity merges (#387):** four **OWNER-only** ops (→ 403 for a PLAYER) — `GET /campaigns/:id/entities/merges` (list; a non-owner sees only `EXECUTED` merges between two `REVEALED` identities), `POST …/entities/merges` (prepare a `PREPARED` "revealed to be" link — validates same-campaign, no self-merge, merged entity not already merged, no cycle via `lib/entity-merges.wouldCreateCycle`), `POST …/entities/merges/:mergeId/execute` (flip `PREPARED`→`EXECUTED` and auto-reveal a `HIDDEN` survivor in one txn, per #379), `DELETE …/entities/merges/:mergeId` (unmerge). A `PREPARED` merge is scrubbed from every non-owner payload (like a `HIDDEN` entity). `backlinks` now **unions** the caller's refs across the entity + every identity that `EXECUTED`-merged transitively into it (`lib/entity-merges.collectMergedInIdentities`, `executedOnly`), each labeled `identity: { id, name }`; a non-owner never sees a `HIDDEN` merged identity's refs/name. Plain-REST; access gated via `assertCampaignMembership`. |
| `routes/campaign-items.ts` | DM-authored campaign items (#380), mounted after `entities`. `GET /campaigns/:id/items` (**OWNER-only** full list incl. `dmNotes` — 403 for a PLAYER), `POST /campaigns/:id/items` (**OWNER-only** create — auto-registers a `HIDDEN` ITEM `CampaignEntity` + `CampaignItemLink` in one txn), `PATCH /campaigns/:id/items/:itemId` (**OWNER-only** update — a rename mirrors onto the fronting entity in the txn; detail blocks upsert), `DELETE /campaigns/:id/items/:itemId` (**OWNER-only**; **cleanup rule:** deletes the item AND its fronting entity in one txn — the entity has no life without its item — mirroring the `CampaignCharacterLink` precedent), `GET /campaigns/:id/items/by-entity/:entityId` (member-readable single item for the Codex card, keyed by the fronting entity: non-owners 404 while that entity is `HIDDEN`, and **`dmNotes` is scrubbed from every player-facing payload**). Category-conditional detail is validated + persisted via the shared `Item*Detail` shapes (`lib/itemDetail.ts`). **Award/revoke (#381):** `POST /campaigns/:id/items/:campaignItemId/award` (`{characterId, quantity?, sessionId?}`) and `POST …/revoke` (`{characterId}`) are **OWNER-only intent-bearing transactions** (`lib/campaign-item-award.ts`) — award snapshots the item + its detail row into a member character's inventory tagged with a `campaignItemId` provenance FK, reveals the fronting entity, and writes an **undoable** `awarded` audit event on the TARGET character; revoke removes the provenance-matched row (`revoked` event). **Session loot (#382):** an explicit `sessionId` threads the loot event onto that session (**400** if it belongs to another campaign or isn't active); omitting it keeps #381's auto-thread of the campaign's active session. Both events stamp `data.recipientName` so the session feed/recap can name who the grant landed on; awarded/revoked net into the session summary's own `loot` line (distinct from `itemsAcquired`). A `isUnique` item already held anywhere in the campaign is a 409 naming the holder; both list + by-entity reads include derived `holders` (`{characterId, characterName, quantity}[]`). Plain-REST for CRUD; access gated via `assertCampaignMembership`. |

`characters.ts` exports `characterInclude` and `serializeCharacter`; every other mutation router imports and calls them to return the same full-character wire shape after applying changes. `POST /characters` is a thin controller: it zod-parses the request, delegates creation to `createCharacter` in `lib/character-create.ts` (returns a discriminated `{ ok, id } | { ok:false, status, error }`), then re-fetches by id with `characterInclude` and serializes — the same persist-then-reserialize idiom the mutation routes use.

### `lib/` — domain logic

*Source of truth: `ls backend/src/lib/`.*

| File | Responsibility |
|---|---|
| `lib/prisma.ts` | Singleton `PrismaClient` with `@prisma/adapter-pg` (required for Prisma 7). Reads `DATABASE_URL`. |
| `lib/config.ts` | Zod-validated, frozen snapshot of the auth-facing env, read once at import. Optional `GOOGLE_CLIENT_ID`/`GOOGLE_CLIENT_SECRET` (a no-creds deploy still boots), `APP_BASE_URL` (default `http://localhost:4000`), tri-state `SESSION_COOKIE_SECURE` (production-default), `ALLOW_DEV_LOGIN` (defaults off; **forced off when `NODE_ENV=production`** regardless of the env), plus pass-through `CORS_ORIGIN`/`SERVE_STATIC_DIR`/`PORT`. `appRedirectUri(provider)` = `${APP_BASE_URL}/api/auth/${provider}/callback`. (Does **not** subsume logger/security/prisma, which read env directly.) |
| `lib/auth/middleware.ts` | `requireAuth` — the gate (mounted in `app.ts` after the health + auth routers). Resolves the `cs_session` cookie via `lookupSession`, attaches `req.user` (typed by `src/types/express.d.ts`), else 401. |
| `lib/auth/access.ts` | `assertCharacterAccess(db, userId, characterId, level)` — the **single chokepoint** every character-scoped route uses instead of ad-hoc `ownerId` checks. Owner-only today (both `view`/`edit`); throws `AuthorizationError` (403) for a non-owner, `NotFoundError` (404) if missing. `level` is the reserved seam for sharing (#116). Accepts a `PrismaClient` or a `$transaction` client. |
| `lib/auth/errors.ts` | `AuthenticationError` (401) / `AuthorizationError` (403) / `NotFoundError` (404) — each carries `.status`, which the terminal `errorHandler` maps straight to the response. |
| `lib/auth/session.ts` | Method-agnostic identity layer: opaque server-side sessions `createSession`/`lookupSession`/`destroySession` over `AuthSession` (the token IS the row id; ~30-day TTL; expired rows rejected + best-effort deleted). `SESSION_COOKIE` = `cs_session`. Any auth method mints sessions the same way. |
| `lib/auth/cookies.ts` | Method-agnostic, hand-rolled cookie handling (no cookie-parser): `parseCookies`/`getCookie`/`serializeCookie` (HttpOnly, SameSite=Lax, Secure-per-config) + `setCookie`/`clearCookie` Response helpers. |
| `lib/auth/oauth/` | The third-party authorization-code **method**, self-contained behind `index.ts` (the seam `routes/auth.ts` imports). `flow.ts` = the provider-agnostic flow (`buildAuthorizeUrl`, `exchangeCode`, `fetchProfile`, the `cs_oauth_tx` state/PKCE codec). `pkce.ts` = PKCE-S256 + state primitives + `OAUTH_TX_COOKIE`. `account.ts` = `resolveUserId` (upsert `AuthAccount`→`User`, link-when-signed-in). `registry.ts` = `enabledProviders()`/`getProvider(id)` reading creds from **live env** (only id+secret-both-set providers enabled). `types.ts` = `ProviderDefinition`/`AuthProvider`/`NormalizedProfile`. `providers/index.ts` is the **manifest** (imports each provider + the array, nothing else); each `providers/<name>.ts` exports one `ProviderDefinition` (descriptor + its own `clientIdEnv`/`clientSecretEnv` + `mapProfile`). **Add a provider = new `providers/<name>.ts` + one array entry + env vars. Add a method (password/magic-link) = a new `lib/auth/<method>/` sibling reusing session + cookies; no OAuth code touched.** |
| `lib/logger.ts` | Pino structured logger + `httpLogger` (pino-http) request-logging middleware. JSON in prod, pretty in dev, silent under test. Level via `LOG_LEVEL`; redacts auth/cookie/password fields. |
| `lib/error-handler.ts` | Terminal Express error middleware (`errorHandler`). Turns uncaught/async route throws into a consistent `{ error }` JSON response; preserves an intentional `status`/`statusCode`; hides 500 detail in prod; logs server-side via the logger. Mounted last in `app.ts`. |
| `lib/security.ts` | `securityHeaders(servesStatic)` (helmet; CSP tuned for the SPA in single-origin mode) + `globalRateLimiter`/`creationRateLimiter` (express-rate-limit, `RATE_LIMIT_*` env knobs, auto-off under test). Mounted high in `app.ts`. |
| `lib/events.ts` | `logEvent(tx, params)` — writes one `CharacterEvent` + per-field `CharacterEventField` diffs inside the caller's transaction. `EventCategory`/`EventType` type unions. |
| `lib/activity.ts` | The audit-log read + undo helpers behind `routes/activity.ts`: `buildActivityQuery(characterId, rawQuery)` (pure query-shaping of the validate-or-ignore filters → `CharacterEventFindManyArgs`, no DB) + `serializeActivityEvent(row)` for the read path, and `revertBatch(db, characterId, batchId)` for LIFO undo. `revertBatch` returns a discriminated `{ ok }` result (`status`/`error` on failure) so HTTP control stays in the route; it owns the 404/409 guards, the reverse-order `$transaction`, and the private per-event `reverseEvent` reversal chain. |
| `lib/srd.ts` | **All 5e rules data**: alignments, skills, ability-modifier math, per-class spellcasting ability, `FULL_CASTER_SLOTS`, `deriveCreatedCharacter()`, `deriveSpellcasting()`, `deriveMulticlassSpellcasting()` (merged caster-level slot pool + separate Warlock Pact Magic; used by `serializeCharacter` for 2+ class entries), `deriveWeaponAttackBonus()`, `deriveWeaponDamage()` (grip-aware: versatile die when off-hand is free), `deriveArmorClassParts()` + `deriveArmorClass()` (labeled AC addends from best equipped body armor + Dex per category + shield, plus Unarmored Defense — Barbarian 10+Dex+Con, Monk 10+Dex+Wis, armor wins; the number is the parts' sum, serialized as `armorClassBreakdown` — AC is derived, never a column), `deriveUnarmoredMovement()` (Monk level-scaled speed bonus +10→+30, zero unless unarmored & unshielded; added to `speed` as a separate additive term in `serializeCharacter`), `deriveFastMovement()` (Barbarian +10 speed at barbarian class level 5+ unless wearing heavy armor, shield irrelevant; a separate additive `speed` term in `serializeCharacter`), `deriveAttacksPerAction()` (weapon attacks per Attack action from Extra Attack: Fighter 2/3/4 at 5/11/20, Barbarian/Monk/Paladin/Ranger 2 at 5, College of Valor bard 2 at 6, else 1 — max across classes for multiclass, never summed; serialized as `attacksPerAction`), `deriveUnarmedStrike()` + `deriveMartialArtsDie()` (unarmed-strike attack/damage; a Monk unarmored & unshielded swaps in max(Dex, Str) and the level-scaled Martial Arts die d4→d10 off the monk class-entry level, else STR + Tavern-Brawler feat die; also flags the strike `magical` at monk class level 6+ (Ki-Empowered Strikes)), `multiclassPrerequisitesMet()` + `MULTICLASS_PREREQUISITES` (ability-score gate for multiclass level-up; thresholds also served on `GET /reference` per class). **This is the only permitted location for rules data.** |
| `lib/class-features.ts` | Class features + trackable resources for all base classes/subclasses (extracted from `srd.ts`). `deriveResources()` — the non-slot analog to `deriveSpellcasting()` (superiority dice, ki, rage). Pure, called inside `serializeCharacter`. `resolveClassDie(source, info)` maps a class-die reference (e.g. `"superiorityDice"`) to its die-face count from the derived pool. |
| `lib/starting-equipment.ts` | `STARTING_EQUIPMENT` per-class packages + choice-group/open-pick structure surfaced via `GET /reference`. (Pack contents themselves are DB-backed; this is the choice scaffolding.) |
| `lib/character-create.ts` | `createCharacter(input, ownerId)` — the `POST /characters` orchestrator. Runs all domain validation (alignment, race/class/subclass catalog anchors, skill/tool-proficiency choices), resolves starting equipment (gold-range check or package group/open-pick validation + `resolveFixedItems` pack expansion + auto-equip), calls `deriveCreatedCharacter`, and persists the row. Returns a discriminated `{ ok:true, id } | { ok:false, status:400, error }` — the route re-fetches + serializes. Persists only; owns no 5e rules of its own (those stay in `srd.ts`/`starting-equipment.ts`). |
| `lib/character-serialize.ts` | `serializeCharacter(row)` — the full character read-model: the derive-on-read pipeline (spellcasting/resources/AC/speed/attacks/proficiencies clamps) mapped to the flat wire shape the frontend consumes. Also `serializeCharacterSummary(row)` for the owner-scoped list. All the private derive helpers (`buildSpellcastingView`, `buildResourcesView`, `applyAdvancementClamp`, `applyFeatLayer`, `buildInventoryContext`, the merged-proficiency + granted-spell builders) live here. Pure; consumed by every mutation router (each re-fetches with `characterInclude` and returns `serializeCharacter(updated)`). |
| `lib/character-include.ts` | `characterInclude` — the shared Prisma `include` for fetching a full character with its race/background/class selections + inventory + journal, plus the derived `CharacterWithRelations` payload type. Imported by every route/lib that reads a full character. |
| `lib/character-schemas.ts` | Shared zod request contracts + inferred types for the character routes: `createCharacterSchema`/`CreateCharacterBody` (consumed by `character-create.ts`) and `updateCharacterSchema` (the PATCH field-patch contract). Keeps the schemas out of `routes/characters.ts` so lib code never imports from `routes/`. |
| `lib/experience.ts` | Pure XP-curve math (no DB): `XP_THRESHOLDS`, `levelForExperience`, `proficiencyBonusForLevel`, `experienceProgress`. |
| `lib/experience-ops.ts` | `applyExperienceOperations()` — transactional XP handler. Also `revertLevelUps()` (auto-reverses HP/dice when XP drops derived level). Calls `reconcileLevelGatedState` after each op. |
| `lib/level-reconciliation.ts` | Level-gated state registry. `reconcileLevelGatedState(ctx)` runs `LEVEL_GATED_RECONCILERS` in order (`reconcileClassEntryLevels` → `reconcileSubclass` → `reconcileManeuvers` → `reconcileDisciplines` → `reconcileToolProficiencies` → `reconcileFightingStyle` → `reconcileAdvancements`) inside the XP transaction. `reconcileClassEntryLevels` trims/removes multiclass `CharacterClassEntry` rows when the derived level drops (emits `class/classLevelsReconciled`). Add new reconcilers here when shipping level-gated features. See `docs/leveling.md`. |
| `lib/advancement.ts` | `applyAdvancementOperations()` — take/remove ASIs and feats; persists the `advancements[]` array (in `resources` JSON) plus the side-effected `abilityScores`/`hitPoints`/`initiativeBonus` columns atomically. |
| `lib/hitpoints.ts` | HP domain: shapes, normalizers, pure rules helpers, `applyHitPointOperations()`. LongRest also resets spell slots in the same transaction. |
| `lib/spellcasting.ts` | `SpellEntry`/`SpellcastingMutableState` shapes, `normalizeSpellcastingMutable()` (handles compact + legacy JSON formats), `applySpellcastingOperations()`. `applyCastSpellOp` is now a thin wrapper over `castAbilityInTx` (below): it maps the spell entry to an `AbilityCost` (`none` for cantrips, `slot` otherwise) + `readEffectSpec`, and the shared caster runs the whole pay-cost → summary → concentration → self-apply sequence. Re-exports `InvalidSpellcastingOperationError` (defined in `ability-cost.ts`). |
| `lib/granted-spells.ts` | 5e rules data for choice-less subclass-granted spells: `deriveGrantedSpells(className, subclass, level)` and `deriveGrantedCastingAbility(subclass)`, driven by the `SUBCLASS_GRANTED_SPELLS` table (each rule carries a `gateLevel` and a `castingAbility`; Way of Shadow → Minor Illusion at L3 via Wisdom). Also exports the `AbilityScores` type (strict 6-key form of `Record<string, number>`). Derived at read time and merged into the spellcasting view — never persisted (`reconcileGrantedSpells` in `level-reconciliation.ts` is the defense-in-depth guard). |
| `lib/effects.ts` | The 5e effect model (dice + save + scaling), extracted from `Spell` so it describes any activated ability. `EffectSpec`/`EffectScaling`/`EffectColumns` types, `readEffectSpec(row, resolveDie?)` (adapter over the flat effect columns; an optional `ClassDieResolver` resolves `dice.faces` from a row's `effectDieSource` class-die reference — e.g. `"superiorityDice"` → the character's fighter-level die — superseding the fixed `effectDiceFaces`), `resolveEffectSpec(spec, effectiveStep, ctx)` → concrete dice `{count,faces,modifier}`, generalizing the scaling axis (`cantripLevel`/`slotUpcast`/`ki`). Adds the `buff` `EffectType` + `resolveBuffSpec(spec)` (adapts a buff effect into a `{target,modifier}` passive-modifier spec). Hand-mirrored on the frontend in `frontend/src/lib/effects.ts` (which returns the real `RollSpec`). |
| `lib/ability-cost.ts` | `AbilityCost` union (`slot`/`pool`/`none`) + the single payer `payAbilityCostInTx(ctx, cost, requested)` — charges a level-N spell slot (with Mystic Arcanum fallback) or a resource pool (delegating to `applySpendResourceInTx`) inside a caller's transaction, returning a `{ label, effectiveStep }`. Owns `InvalidSpellcastingOperationError`. `readAbilityCost(row)` adapts flat cost columns (GrantedAbility) into the union, mirroring `readEffectSpec`. One-directional graph: spellcasting → ability-cost → resources. |
| `lib/ability-cast.ts` | `castAbilityInTx(ctx, input)` — the one shared caster behind `castSpell` (and a future `castDiscipline`): pay-cost (via `payAbilityCostInTx`) → build summary → drop/set concentration → self-apply → return the `OpOutcome` the dispatcher logs. Concentration displace and self-apply are lifted here as character-wide helpers operating on the mutable state's `concentratingOn` (not a `SpellEntry`), so a non-spellcaster ability can hold concentration too. The `effect` (`EffectSpec`) only shapes the summary — the client roll is trusted, never bound-checked. One-directional graph: spellcasting → ability-cast → {ability-cost, effects, spell-state, hitpoints, events}. |
| `lib/resources.ts` | `applyResourceOperations()` — spend/restore class resources, learn/forget maneuvers + tool profs, and learn/forget/swap Four Elements disciplines (GrantedAbility catalog, `source: "discipline"`); persists `used` counts and known lists in `resources` JSON. Analog to `spellcasting.ts`. Discipline learn/swap enforce the derived cap + per-discipline min monk level; swap is gated to one per monk level via `DisciplineEntry.lastSwapLevel`. `applySpendResourceInTx` returns its `ResourceOpAudit` so callers (`ability-cost.ts`) can surface the spend label. |

## Cross-cutting data patterns

### Catalog + per-character row

Two distinct approaches, both in play:

**Selection tables** (race/class/background): a nullable FK to the catalog *and* an own `name` snapshot. The character's displayed race/class/bg can drift from the catalog (homebrew, renamed) — the snapshot is the source of truth, the FK is just provenance. `serializeCharacter` flattens these back to the flat wire shape.

**Full snapshots** (inventory items, spells): at acquire/learn time, all catalog fields are copied into the per-character row (`InventoryItem`, spell `SpellEntry` in the JSON). After that, the catalog is ignored — the snapshot is fully self-contained and freely editable (e.g. "Club" → "Club +1"). No merge-with-catalog logic anywhere.

**Detail tables** (items only): `ItemWeaponDetail`/`ItemArmorDetail`/`ItemConsumableDetail` + their `InventoryWeapon/Armor/ConsumableDetail` snapshot mirrors. Spells are flat — school/level/effectKind/effectDice/scaling all on the `Spell` row directly, no detail table split needed.

**Dice fields**: decomposed as `...DiceCount`/`...DiceFaces`/`...Modifier` to match `dice.ts`'s `RollSpec` shape — never stored as `"1d6"` strings.

See `schema.prisma` model comments for the detailed snapshot-vs-overlay reasoning.

### Identity & ownership

*Source of truth: `User` / `AuthAccount` / `AuthSession` + `Character.ownerId` in `schema.prisma`.*

- `User` — identity row: `email? @unique`, `name?`, `imageUrl?`. Holds no tokens.
- `AuthAccount` — one row per linked sign-in method, `@@unique([provider, providerAccountId])`. OAuth/credential **tokens live here, never on `User`** (`accessToken`/`refreshToken`/`idToken`/…); `passwordHash` is future-credentials-only and null for OAuth.
- `AuthSession` — server-side session whose `id` **is the opaque cookie token** (no `@default` — generated by the auth layer and stored verbatim).
- `Character.ownerId` — required FK to `User` (`onDelete: Cascade`, `@@index`). Emitted by `serializeCharacter`/`serializeCharacterSummary`.

**Shared campaigns (#246):** `Campaign` (name, `ownerId` → `User` relation `"CampaignOwner"`, unique `inviteCode`, `createdAt`/`updatedAt`, `characters Character[]`) and `CampaignMembership` (`@@unique([campaignId, userId])`, `role: CampaignRole = OWNER|PLAYER`, `joinedAt`/`updatedAt`) add a shared backbone. `Character.campaignId` is a nullable FK (`onDelete: SetNull`, `@@index`) emitted by `serializeCharacter`. `User` gains back-relations `ownedCampaigns` + `campaignMemberships`. Access is gated by `assertCampaignMembership(db, userId, campaignId, level)` in `lib/auth/access.ts` — the campaign mirror of `assertCharacterAccess` (404 missing / 403 non-member). Plain-REST domain (no audit log).

**Campaign entity registry & @-tagging (#248):** `CampaignEntity` (`campaignId` → `Campaign` `onDelete: Cascade`, `type: EntityType = NPC|LOCATION|FACTION|ITEM|PC|OTHER`, `name`, `aliases String[]`, nullable `notes`, `visibility: EntityVisibility = HIDDEN|REVEALED` default `REVEALED` (#379), `@@index([campaignId, type])`) is a fully-polymorphic shared wiki — NO per-type FK columns. `visibility` is the DM reveal state: non-owners only ever see `REVEALED` entities (list/detail/backlinks/mentions), the owner sees all; the backfill migration + auto-created PC entities are `REVEALED`. `JournalEntryRef` (`@@unique([entryId, entityId])`, `@@index([entityId])`, cascades from both `JournalEntry` and `CampaignEntity`) materializes one backlink per @-tag, derived from `@[<uuid>]` tokens in the entry body on journal write (`lib/journal-refs.ts`). `CampaignCharacterLink` (1:1, both `@unique`) ties a character to its auto-created PC entity (made on attach). `CampaignEntityMerge` (#387; `campaignId` → `Campaign` `onDelete: Cascade`, `mergedEntityId @unique` → the old identity, `survivorEntityId` → the true identity, both cascade from `CampaignEntity` so the record dies with either side, `status: MergeStatus = PREPARED|EXECUTED` default `PREPARED`, nullable `note`, `preparedAt`, nullable `executedAt`, `@@index([campaignId])`+`@@index([survivorEntityId])`) is a non-destructive "revealed to be" link — no row is destroyed and no `JournalEntryRef` is rewritten, so a chip that tagged the old id keeps resolving to it. `PREPARED` is the DM's secret prep (scrubbed from non-owner payloads like a `HIDDEN` entity); `EXECUTED` is the public reveal. Chains resolve transitively (Jenkins→Vecna→Whispered One) via the pure `lib/entity-merges.ts` graph helpers. Back-relations: `CampaignEntity.mergedInto`/`survivorOf`, `Campaign.entityMerges`, `Campaign.entities`, `JournalEntry.refs`, `Character.campaignLink`. Plain-REST domain (no audit log); access gated via `assertCampaignMembership`.

**DM-authored campaign items (#380):** `CampaignItem` (`campaignId` → `Campaign` `onDelete: Cascade`, `name`, nullable `description`, `category: ItemCategory`, nullable `rarity`, `requiresAttunement`/`isUnique` bools, nullable `weight`/`cost` (Json, same `{cp,sp,gp,pp}` shape as `Item.cost`), nullable **`dmNotes`**, `@@index([campaignId])`) is a DM's loot/magic-item prep that lives in a campaign, NOT on any sheet. Per-category mechanics live on the optional 1:1 `CampaignItemWeaponDetail`/`CampaignItemArmorDetail`/`CampaignItemConsumableDetail` tables (same shapes as the catalog `Item*Detail`, `onDelete: Cascade`), not as nullable columns. `CampaignItemLink` (1:1, both `@unique`, cascades from both sides) fronts each item with an auto-created `HIDDEN` ITEM `CampaignEntity` — so it's `@`-mentionable and gets reveal/hide + backlinks for free from the #248/#379 machinery, **with zero new mention code**. Two invariants live in `routes/campaign-items.ts`: (1) `dmNotes` is DM-private and is serialized only into owner payloads (scrubbed from the member-readable by-entity read); (2) the **delete cleanup rule** — deleting an item also deletes its fronting entity in the same transaction (the entity has no life without its item), mirroring the `CampaignCharacterLink` precedent. Back-relations: `Campaign.items`, `CampaignEntity.itemLink`.

All three auth children cascade-delete with their `User`. **Ownership is enforced (#101):** `requireAuth` (`lib/auth/middleware.ts`) attaches `req.user` from the session; `POST /characters` stamps `ownerId = req.user.id`; `GET /characters` is scoped to `req.user`; and every character-scoped route resolves access through `assertCharacterAccess` (`lib/auth/access.ts`) — owner-only today, 403 for a non-owner, 404 for a missing character, with `level: "view" | "edit"` as the single seam where sharing (#116) will widen. There is no bootstrap-owner fallback; a caller must be signed in. (The migration that introduced `ownerId` backfilled pre-existing characters onto one minted user before enforcing `NOT NULL`.)

### JSON columns on Character

*Source of truth: Character model in `schema.prisma`.* (Journal is **not** a JSON column — it's a separate `JournalEntry` table: `kind` (`NOTE` | `ENTRY`), `date`, `loggedAt` (capture timestamp), `body`, `visibility` (`PRIVATE` | `CAMPAIGN`, default `PRIVATE`), and a denormalized `authorUserId`. The `add_journal_capture_fields` migration backfilled `authorUserId` from `character.ownerId` before enforcing `NOT NULL`. Surfaced under `serializeCharacter().journal`, newest-first by `date` then `loggedAt`.)

The Character row carries these JSON columns: `hitPoints`, `hitDice`, `abilityScores`, `skills`, `toolProficiencies`, `currency`, `spellcasting?`, `resources?`, `conditions?`, `activeEffects?`. Notable ones:

- `spellcasting Json?` — mutable state only: `{ slotsUsed, arcanumUsed, spells: SpellEntry[], concentratingOn }`. Slot totals/DC/attack/ability are derived at read time by `deriveSpellcasting()` in `serializeCharacter` (single class); a multiclass character (2+ entries) instead derives via `deriveMulticlassSpellcasting()`, which merges caster levels into one slot pool and surfaces Warlock Pact Magic in a separate `spellcasting.pact` field (single-class output is unchanged). Choice-less subclass-granted spells (`SpellEntry.source === "subclass"`, e.g. a Way of Shadow monk's Minor Illusion) are also **derived** and merged into the read view by `deriveGrantedSpells()` (`lib/granted-spells.ts`) — never persisted; the learned copy wins on a name collision. `normalizeSpellcastingMutable()` handles legacy blobs. A cast Shadow Art holds concentration under a `concentratingOn.entryId` prefixed with `SHADOW_ART_CONCENTRATION_PREFIX` (`"shadow-art:"`, `lib/shadow-arts.ts`) so its id space is disjoint from any `Spell.id`; the `resolveConcentration` clamp-on-read keeps an entry only when it's a current spellbook spell **or** carries that prefix (a forgotten spellbook spell is dropped even for a Way of Shadow monk).
- `resources Json?` — mutable state only: `{ used, maneuversKnown, disciplinesKnown, toolProficienciesKnown, advancements, fightingStyle }`. Pool totals/die/recharge and all level-gated caps are derived at read time via `deriveResources()`; the persisted lists are clamped-on-read. `disciplinesKnown[].disciplineId` stores a GrantedAbility catalog uuid (`source: "discipline"`) — unchanged by the table rename. `maneuversKnown[].maneuverId` likewise points at a GrantedAbility uuid (`source: "maneuver"`) with a `placement`/`actionSlot` snapshot for session-UI routing (custom maneuvers carry `maneuverId=null` and spend die-only). The retired `Maneuver` table's rows seeded these catalog entries; a data migration name-matched existing `maneuversKnown` entries onto the new uuids.
- `conditions Json?` — `{ active: ConditionEntry[], exhaustion: number }`. Pure mutable state, not level-derived.
- `activeEffects Json?` — `{ buffs: ActiveBuff[] }` (`lib/active-effects.ts`). Passive stat modifiers, deduped by `key` on re-apply. Each carries a `duration` axis (#455): `concentration` (the #438 default) clears when its `sourceEntryId` concentration ends (`clearBuffsForSourceInTx`, gated to concentration-duration only); `while-active` persists until toggled off (`clearBuffByKeyInTx`) or ended by the session turn-hook (below); `until-rest` clears on the matching rest (`clearBuffsForRestInTx` — long rest clears both `short` and `long` `restType`). The latter two are durable self-buffs (Rage #457/#458), activated independent of a cast and unaffected by concentration changes. A long rest or falling to 0 HP additionally ends **all** `while-active` buffs via `clearWhileActiveBuffsInTx` (wired into the `longRest`/`damage` HP ops). Rage itself is applied by the `rage` action (a `while-active` `meleeDamage` buff scaled by barbarian level) and cleared by the `endRage` action; both flow through the actions orchestrator's `applyBuff`/`clearBuff` ops. Serialization omits `duration`/`restType` for concentration buffs, keeping #438 buffs byte-identical. Concentration buffs are seeded by `castAbilityInTx` via `resolveBuffSpec` (`lib/effects.ts`). `serializeCharacter` sums each target: skill/ability targets become an optional `tempModifier` + `tempModifierSources` breakdown; the `meleeDamage` target is threaded into `deriveWeaponDamage` and added to melee weapons' `damageModifier` (ranged untouched), so the attack/damage engine reads the buffed damage via `w.damage`.
- `currency Json` — the **only** JSON column still patchable via `PATCH /characters/:id`. Every other column above mutates exclusively through its domain's `…/transactions` endpoint.

`armorClass` is **not** a column: `serializeCharacter` derives it at read time via `deriveArmorClassParts()` from the best equipped body-armor `InventoryItem` snapshot + effective Dex (capped per armor category) + shield, then appends Defense fighting-style and feat `armorClass` parts. While unarmored, Unarmored Defense applies from the class entries (Barbarian 10+Dex+Con, shield stacks; Monk 10+Dex+Wis, unusable while wielding a shield; highest formula wins, worn armor overrides). The serialized character carries both the number and its provenance: `armorClassBreakdown` is an ordered `{ label, value }[]` (e.g. `"Half Plate"`, `"Dex (max +2)"`, `"Shield"`, `"Unarmored Defense"`, `"Feats"`) whose values sum exactly to `armorClass` — the frontend renders the labels verbatim and never does AC math. **Ordering contract:** the base part (armor name / `"Unarmored"` / `"Unarmored Defense"`) is always first — the frontend renders index 0 as a plain number and every later part as a signed modifier, so new bonus sources must be appended, never prepended. Equip/unequip armor (or change Dex) via the inventory/ability transactions and AC re-derives; it's absent from the `PATCH /characters/:id` allowlist.

### Unified audit log

`CharacterEvent` + `CharacterEventField` in `schema.prisma`:

- **Single-Table Inheritance**: `category` + `type` discriminators. Full `EventCategory` set (source of truth: `lib/events.ts`): `inventory`, `hitPoints`, `experience`, `currency`, `spellcasting`, `class`, `resources`, `advancement`, `session`, `combat`, `conditions`, `effects` (active buffs; `activity.ts` revert restores `before.activeEffects`), `roll`.
- **`roll` category** houses every die roll logged from the session UI — `attackRoll`, `damageRoll` (re-homed here from `combat` in #128), `checkRoll`, `saveRoll`, `initiativeRoll`. Written by `logRollEvent` (`lib/sessions.ts`) via the character-scoped `…/roll` route; `data` carries `{ kind, source, total, specLabel, damageType, faces, ability, skill, dc, rollMode }`. Roll events are **non-undoable**: `before`/`after` are `null` and the whole `roll` category is excluded from the LIFO guard (below), so a roll neither undoes itself nor blocks undo of a real action.
- **`inventory` category** covers player-driven ops (`acquired`/`consumed`/`sold`/`bought`/`removed`/`equipped`/`unequipped`) plus DM item award/revoke (#381): `awarded` (creates the row — `before:null`) and `revoked` (deletes it — stashes `data.deletedItem`) are written by `lib/campaign-item-award.ts` on the TARGET character with `actor:"dm"`, and revert through the same shape-driven `revertInventoryEvent` (delete-created / recreate-deleted). The `deletedItem` snapshot carries `campaignItemId` alongside `itemId` so recreate-on-undo restores the campaign-item provenance FK (both nulled if the referenced row was since deleted) — without it, an undone revoke would fall out of the holder + unique-guard queries.
- **Polymorphic soft-reference**: `entityType`/`entityId` (no FK — the entity may be deleted).
- **before/after JSON snapshots**: the state before and after the operation, used by the revert handler to restore. The free-form `data` JSON carries op-specific extras the revert handler also reads — e.g. inventory delete ops stash a self-contained `data.deletedItem` snapshot (all scalar columns + the weapon/armor/consumable detail block) so the deleted row can be rebuilt on undo. (`data` lives outside `before`/`after` precisely so it is never fed to `diffToFields`.)
- **Append-only**: events are flagged `reverted:true`, never deleted. A `revert` meta-event is appended on undo.
- **`class` category** carries multiclass structural changes: `classLevelsReconciled` (level-down trims/removes `CharacterClassEntry` rows) reverts via a custom branch in `activity.ts` that **recreates** any entries deleted during reconcile; `classAdded` (the `addClass` op) and a multiclass "new class" level-up both stash `data.createdClassEntryId` so undo can delete the entry they created — `classAdded` also restores the `hitPoints`/`hitDice` bump from its `before` snapshot.
- **Session tagging**: `sessionId String?` — events fired while a session is active get its id; between-session events (shopping, leveling between adventures) get `null`. Events stay **per-character** (`CharacterEvent.characterId` is unchanged); a campaign session aggregates them by `sessionId`. `getActiveSessionId(characterId)` (called at the top of every `apply*Operations()`) now resolves the character's `campaignId` → the campaign's active session — so a character with no campaign always tags `null`.

`logEvent(tx, params)` in `lib/events.ts` writes the event + computes `CharacterEventField` diffs (via `diffToFields`) inside the caller's `$transaction`. All ops in a single request share a `randomUUID()` `batchId`.

Read: `GET /characters/:id/activity` returns the chronological (newest-first) stream. The route handler is a thin controller over `buildActivityQuery` + `serializeActivityEvent` (`lib/activity.ts`). Optional, composable (AND) query params — all validate-or-ignore (an unknown enum value is silently dropped, never a 400): `?category=<CharacterEventCategory>`, `?type=<CharacterEventType>`, `?sessionId=<id>` (per-session view), `?entityId=<id>` (per-entity view, e.g. one InventoryItem), `?includeFields=1` (attach the per-field diff rows), `?reverted=0|1` (exclude/include reverted events; default: all). The frontend `ActivityModal` drives `category`/`type`/`sessionId`/`entityId` from its filter bar.

Undo: `POST /characters/:id/events/:batchId/revert`; the handler is a thin controller over `revertBatch` (`lib/activity.ts`), then re-fetches + returns the serialized character. LIFO-only (returns 409 if not the most-recent non-reverted batch). Restores `before` snapshots in a transaction. Inventory events are fully revertable: `revertInventoryEvent` (`lib/inventory.ts`) reconstructs deleted `InventoryItem` + detail rows from `data.deletedItem` (reusing the original id) and reverses currency from `data.currencyDelta` — handled at the top of the revert loop before the `if (!before) continue` guard, since an `acquire` event carries `before == null` and undoing it deletes the created row. **The LIFO guard skips events from ended sessions** (`OR: [{ sessionId: null }, { session: { status: "active" } }]`) — a closed session's history is frozen and cannot be undone. It also skips the whole `roll` category (`category: { not: "roll" }`) so non-undoable roll log entries never sit at the head of the LIFO stack.

### Intent-bearing transaction pattern

Every mutable domain follows the same shape:

1. **Zod discriminated-union** per op type in the route (e.g. `castSpell | expendSlot | …`)
2. **`apply*Operations(characterId, ops)`** in `lib/` — one `prisma.$transaction`, applies ops in order, calls `logEvent` per meaningful op with shared `batchId`
3. **Route** calls the lib function, catches domain errors → 400, re-fetches and returns `serializeCharacter`

`lib/inventory.ts` is the reference implementation. Do not add new mutable domains via `PATCH /characters/:id`.

## Docker Compose

Four services: `db` (Postgres 17, port 5432), `pgadmin` (port 5050), `backend` (Express, port 4000), `frontend` (Vite, port 5173). Each of backend/frontend has its own `Dockerfile` with source bind-mounted for hot reload and an anonymous volume shadowing `node_modules`. No shared root Dockerfile — they're independent npm packages; the root workspaces setup is for local fanning-out only, not containers.
